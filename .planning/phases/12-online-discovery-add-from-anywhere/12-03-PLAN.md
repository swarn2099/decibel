<task type="auto">
  <name>Task 1: Recommendations API endpoint</name>
  <files>src/app/api/passport/recommendations/route.ts</files>
  <action>
Create `src/app/api/passport/recommendations/route.ts` (GET endpoint):

1. Requires auth (createSupabaseServer to check user). Get fan by email from fans table.

2. Build recommendation logic:
   a. Get fan's collected performer IDs (from collections table where fan_id matches).
   b. Get genres from those performers — build a frequency map of genres.
   c. Get top 3-5 genres by frequency.
   d. Query performers table for artists NOT in fan's collected set, WHERE genres overlap with fan's top genres. Use Supabase `.overlaps()` filter on the genres array column.
   e. Order by: performers with upcoming events first (LEFT JOIN events WHERE event_date >= today, count > 0 first), then by follower_count DESC as tiebreaker.
   f. Limit to 10 recommendations.

3. For each recommended performer, also check for upcoming events:
   - Query events table for performer_id where event_date >= today, join venues for name.
   - Include `next_show: { venue_name, event_date } | null` in response.

4. Return response:
```typescript
{
  recommendations: Array<{
    id: string;
    name: string;
    slug: string;
    photo_url: string | null;
    genres: string[];
    city: string | null;
    match_reason: string; // e.g., "Based on your love for tech house" or "Similar to artists you've collected"
    next_show: { venue_name: string; event_date: string } | null;
  }>;
  based_on_genres: string[]; // top genres used for matching
}
```

5. Handle edge case: if fan has no collections, return empty recommendations with a helpful message. If fan has collections but no genre overlap found, fall back to popular performers (highest follower_count) not yet collected.

6. Use createSupabaseAdmin for DB reads (service role to avoid RLS issues).
