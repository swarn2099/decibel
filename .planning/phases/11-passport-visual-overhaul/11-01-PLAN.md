**A. Create src/app/passport/page.tsx (server component):**

Server component that fetches all passport data. Auth-gated (redirect to /auth/login if no user).

1. Get auth user via createSupabaseServer
2. Find fan by email via createSupabaseAdmin
3. Query collections table joined with performers and venues, ordered by created_at DESC. Include ALL collections (both verified=true and verified=false). This is the timeline data.
4. Query fan_tiers for tier info per performer (left join conceptually — map tiers onto timeline entries by performer_id)
5. Pass data to PassportClient as typed props using the PassportTimelineEntry[] and PassportFan types

Fan slug generation: derive from fan name (slugify: lowercase, replace spaces with hyphens, remove special chars) or fall back to first 8 chars of fan.id. Do NOT require a DB migration — compute at query time and pass as prop. This slug is used for the "share" URL display only in this plan.

**C. Create src/app/api/passport/stats/route.ts:**

GET endpoint, auth-gated. Computes PassportStats server-side using admin client:

1. Find fan by auth user email
2. Count verified collections (verified=true), group by performer_id for unique artists
3. Count discovered collections (verified=false), group by performer_id for unique discoveries
4. Count unique venues from verified collections
5. Count unique cities from performers in verified collections
6. Find favorite genre: collect all genres arrays from performers in verified collections, count frequency, return most common
7. Find most collected artist: group verified collections by performer_id, count, return max
8. Find most visited venue: group verified collections by venue_id, count, return max
9. Calculate streak: get all verified collection dates ordered DESC, count consecutive weeks (a week counts if any scan exists in that Mon-Sun period)
10. Return PassportStats JSON

Use raw SQL via supabase.rpc() or multiple queries — keep it efficient but don't over-optimize. Server-side only.
