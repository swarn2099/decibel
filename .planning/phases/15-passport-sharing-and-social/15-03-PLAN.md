```typescript
export type ActivityType = 'collection' | 'discovery' | 'badge';

export interface ActivityFeedItem {
  id: string;
  type: ActivityType;
  fan: { id: string; name: string; slug: string };
  // For collection/discovery:
  performer?: { id: string; name: string; slug: string; photo_url: string | null };
  venue?: { name: string } | null;
  // For badge:
  badge?: { id: string; name: string; icon: string; rarity: string };
  created_at: string;
}

export interface ContactCheckResult {
  email: string;
  fan: { id: string; name: string; slug: string } | null;
  isFollowing: boolean;
}
```

**Activity feed endpoint** (`GET /api/social/feed`):
- Auth required (get current user, find fan by email)
- Query fan_follows to get list of following_ids for current fan
- For each followed fan, check their fan_privacy setting:
  - 'public': include their activity
  - 'mutual': include only if they also follow current fan
  - 'private': exclude entirely
- Fetch recent activity (last 30 days, limit 50) by querying:
  1. Collections from visible followed fans: `collections` table where `fan_id in (visible_ids)`, join with `performers` and `venues`, ordered by created_at desc
  2. Badges from visible followed fans: `fan_badges` table where `fan_id in (visible_ids)`, ordered by earned_at desc
- Merge and sort all results by timestamp descending
- For each collection, determine if it's a 'collection' (verified=true) or 'discovery' (verified=false)
- For each badge, look up definition from BADGE_DEFINITIONS to get name and icon
- Include fan name and slug for each activity item (for linking to their passport)
- Use generateFanSlug from @/lib/fan-slug for fan slugs
- Return `{ items: ActivityFeedItem[], hasMore: boolean }`
- Support `?cursor=ISO_DATE` for pagination (fetch items older than cursor)
- Use admin client for all DB queries

**Contact-check endpoint** (`POST /api/social/contact-check`):
- Auth required
- Body: `{ emails: string[] }` (max 500 emails)
- Validate: array of strings, max 500, each must look like an email
- Query fans table for matching emails
- For each match, check if current fan already follows them
- Return `{ matches: ContactCheckResult[] }` -- only return entries that matched a fan
- This is the web-compatible version of SOCL-06. Full phone contact sync requires mobile app (MOBILE-04), but email-based contact matching works on web.
- NOTE: Do not send notifications in this plan. The "someone you know joined" notification is noted as mobile-dependent (MOBILE-03/04). This endpoint enables the web-side discovery of contacts already on the platform.
